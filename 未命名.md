要求如下：

  1. 目标收敛成一个最小闭环：给定一个 task_id，worker 能自动把任务从 ready/running 推进到下一个稳定状态，也就是running、waiting_on_control、paused、completed、failed 之一。
  2. 严格按文档定义步序实现，不要自创流程：
      - Load task
      - Apply pending control signals
      - Assemble working state
      - Decide next action
      - 如需 capability，则先 policy evaluate、再 validate input、再 dispatch capability
      - Commit result into state/evidence
      - Emit events
      - Optionally create checkpoint
      - Decide next status
      - 这是 minimal-reference-runtime-v0.1.md:155 已经给出的主线，不要改成“边聊边拼”的 agent loop。
  3. 模型输出先限制成极小的结构化动作集，不要开放自然语言自由发挥。建议只允许四类：
      - invoke_capability
      - complete_task
      - fail_task
      - update_working_state
      - 模型只能提出动作意图，不能直接越过 capability boundary 做事。
  4. 这一步只支持极少数 demo capability，按参考运行时建议收敛到 3 个左右即可：
      - fs.read
      - fs.write
      - exec.run
      - 文档推荐的最小能力集合已经给了方向 minimal-reference-runtime-v0.1.md:182。不要一开始做浏览器、网络、记忆、插件市场。
  5. approval 不是单独旁路，必须走同一条 step loop。worker 碰到需要审批的 side effect 时，任务进入 waiting_on_control；收到 approve/deny 后，再由下一次 AdvanceTaskStep 继续推进。不要做隐藏状态机。
  6. 这一步的完成标准必须是可演示，而不是“接口差不多”：
      - 一个新任务能自动跑起来。
      - 它能至少做一次模型决策。
      - 它能至少调用一次 capability。
      - 它能在需要审批时停住。
      - 它能在 approve 后继续跑到完成。
      - 全程都能在事件流里解释清楚。
  7. 这一步明确禁止扩 scope：
      - 不做多 agent。
      - 不做 streaming UI。
      - 不做复杂 memory。
      - 不做 marketplace。
      - 不新增 kernel primitive。
      - 不先追求“通用智能”，只追求“一个可解释、可推进、可中断、可恢复的 demo loop”。
